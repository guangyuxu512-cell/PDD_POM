## Codex 任务：精简售后任务 — 仅保留扫描 + 写入队列

### 背景

`tasks/售后任务.py` 当前逻辑过于复杂，一次性做了扫描、详情抓取、决策、按钮点击、弹窗处理、飞书通知等所有事情，导致调试困难。需要先把链路拆开，做到最小闭环稳定后再逐步恢复。

### 目标

重构 `tasks/售后任务.py`，**仅保留以下最小闭环**：

1. 导航到售后列表页（`售后页.导航并拦截售后列表()`）
2. 点击"待商家处理"卡片（已包含在上述方法中）
3. 抓取列表数据（API 拦截优先，DOM fallback 兜底）
4. **按订单号去重后**写入 SQLite 队列
5. 自动翻页，重复 3-4，直到无下一页

### 具体要求

#### 1. `tasks/售后任务.py` 修改

- **移除**所有详情页处理逻辑（`_处理单条`、`_处理扫描结果页`、`_执行详情页转人工`、`_执行DOM扫描回退` 中的处理部分）
- **移除**决策引擎调用（`售后决策引擎` 的 import 和使用）
- **移除**飞书通知（`飞书服务` 的 import 和使用）
- **移除**售后配置服务的 import 和使用（`售后配置服务`）
- **移除**详情页相关操作（打开详情标签、抓取详情、截图、关闭标签等）
- **移除** `_执行结果` 和 `_售后配置缓存` 字段
- **保留** `售后队列服务` 的使用
- **保留** `售后页` POM 的使用（仅用到：`导航并拦截售后列表`、`获取售后单数量`、`获取第N行信息`、`翻页并拦截`）
- **保留** `基础任务` 基类继承和 `@register_task` 装饰器
- **保留** `@自动回调("售后处理")` 装饰器在 `执行` 方法上

#### 2. 售后类型标准化

新增一个静态方法 `_判断售后类型(原始类型: str) -> str`，将拼多多返回的原始售后类型归类为 5 种标准类型：

- `"退货退款"` — 原始类型同时包含"退货"和"退款"
- `"退款"` — 原始类型包含"退款"（不含"退货"）
- `"补寄"` — 原始类型包含"补寄"
- `"换货"` — 原始类型包含"换货"
- `"维修"` — 原始类型包含"维修"
- 无法归类时保留原始值

补寄、换货、维修统一在队列记录中标记 `需要人工 = True`。

#### 3. 队列记录结构

`_构建队列记录` 方法生成的 dict 包含以下字段：

```python
{
    "batch_id": 批次ID,
    "shop_id": 店铺ID,
    "shop_name": 店铺名称,
    "订单号": 摘要.get("订单号", ""),
    "售后类型": 标准化后的类型,
    "售后类型_原始": 原始类型文本,
    "售后状态": 摘要.get("售后状态", ""),
    "退款金额": 提取后的float金额,
    "商品名称": 摘要.get("商品名称", ""),
    "需要人工": bool,
}
```

#### 4. 订单号去重（关键）

在调用 `售后队列服务.批量写入队列` 之前，或者在 `批量写入队列` 方法内部，**必须按订单号去重**：

- 检查 `售后队列服务.py` 中的 `批量写入队列` 方法，确认是否已有去重逻辑
- 如果没有，在写入前用 SQL `INSERT OR IGNORE` 或先 `SELECT` 检查已存在的订单号，跳过已存在的记录
- 去重维度：同一个 `订单号` 在队列中只应存在一条记录（不区分批次）
- 确保这个去重逻辑不会和 `售后队列服务.py` 中已有的其他方法（如 `标记已完成`、`标记人工`、`更新阶段` 等）产生冲突

#### 5. 执行方法的简化流程

```
执行(页面, 店铺配置):
  1. 创建批次 → 队列服务.创建批次(店铺ID)
  2. 导航+拦截 → 售后页.导航并拦截售后列表()
  3. 循环每页:
     a. 如果API拦截数据为空 → fallback到DOM抓取(获取售后单数量 + 逐行获取第N行信息)
     b. 如果仍为空 → break
     c. 遍历当前页数据，构建队列记录列表
     d. 批量写入队列（按订单号去重）
     e. 翻页 → 售后页.翻页并拦截()，返回None则break
  4. 上报统计汇总
```

#### 7. 可以修改的文件

- `tasks/售后任务.py` — 主要重构目标
