### 背景

售后扫描任务的 API 拦截时序有问题，导致：

1. 导航到售后列表页时，拦截器提前捕获了"全部"状态的默认列表数据
2. 点击"待商家处理"后触发的真正请求反而拦不到
3. 翻页后的请求也经常拦不到
4. 最终大量依赖 DOM fallback 逐行抓取兜底，效率低、日志刷屏

### 实际运行日志（体现问题）

```
[售后页] 售后列表页加载完成
[售后页] API拦截抓取到 10 条售后单        ← 错误！导航触发的默认请求被捕获了
[售后页] 已点击待商家处理卡片
[售后页] API拦截抓取到 0 条售后单         ← 真正的请求没拦到
[售后页] 第1行~第10行 DOM逐行抓取          ← 被迫fallback
[售后页] API拦截抓取到 0 条售后单         ← 翻页后也没拦到
[售后页] 第1行~第10行 DOM逐行抓取          ← 又被迫fallback
[售后页] API拦截抓取到 4 条售后单         ← 偶尔拦到
```

### 预期修复后的日志

```
[售后页] 售后列表页加载完成
[售后页] 等待默认请求完成
[售后页] 已点击待商家处理卡片
[售后页] API拦截抓取到 10 条售后单        ← 正确：待商家处理的数据
[售后页] API拦截抓取到 4 条售后单         ← 翻页后的数据
```

### 需要修复的方法

#### 1) `导航并拦截售后列表` 方法

**当前有问题的时序**：

```python
async def 导航并拦截售后列表(self):
    await self.导航到售后列表()
    # 此处导航已触发默认列表API请求
    拦截任务 = asyncio.create_task(self.拦截售后列表API(超时秒=10))
    # 拦截器可能已经捕获了导航时的默认请求数据！
    await self.确保待商家处理已选中(强制点击=True)
    结果 = await 拦截任务
    ...
```

**修复为以下时序**：

```
1. 导航到售后列表页
2. 等待页面完全加载
3. 额外 await asyncio.sleep(2)，确保导航触发的默认列表请求已经返回完毕
4. 此时才创建拦截任务（注册 response 监听器）
5. 然后点击"待商家处理"卡片（触发新的列表请求）
6. 等待拦截任务捕获这次新请求
```

这样拦截器注册时，导航的默认请求早已完成，response 事件已经过去了，不会被误捕获。

**重试逻辑也要调整**：如果第一次拦截为空，重试时同样要确保先注册拦截再触发请求（强制再次点击待商家处理卡片）。

#### 2) `拦截售后列表API` 方法

检查以下潜在问题：

- `捕获事件 = asyncio.Event()` 只能 set 一次。如果被残留的响应提前 set 了，后续真正的响应就被忽略了。确保每次调用 `拦截售后列表API` 时，`捕获事件` 是全新的。
- `结果容器.clear()` 在回调中被调用，如果有多个响应并发到达，可能存在竞态：先 clear 了，新数据还没填进去，但 `捕获事件` 已经被之前的响应 set 了。
- 确保 `_处理响应` 回调中，只有当 `结果容器` 非空时才 `捕获事件.set()`。

#### 3) `翻页并拦截` 方法

当前逻辑：

```python
async def 翻页并拦截(self):
    if not await self._检查有下一页():
        return None
    拦截任务 = asyncio.create_task(self.拦截售后列表API(超时秒=10))
    翻页成功 = await self.翻页()
    if not 翻页成功:
        拦截任务.cancel()
        return None
    return await 拦截任务
```

这个时序本身是对的（先注册拦截器再翻页），但如果 `拦截售后列表API` 内部有状态残留问题，也会导致拦截失败。修复 `拦截售后列表API` 的问题后，这个方法应该就能正常工作。

如果仍然不稳定，可以在 `翻页()` 之后增加一个小延迟（0.5~1秒），给网络请求一点缓冲时间。

### 验收标准

- [ ] 导航到售后列表页后，不会立即打印 "API拦截抓取到 N 条"
- [ ] "API拦截抓取到 N 条" 出现在 "已点击待商家处理卡片" **之后**
- [ ] 第一次拦截到的数据确实是"待商家处理"状态的售后单
- [ ] 翻页后 API 拦截能稳定拿到下一页数据
- [ ] 极少甚至不需要 fallback 到 DOM 逐行抓取
- [ ] 共 14 条待处理售后单时，扫描总数应为 14（不是 34）
