# 流程执行架构改造方案

## 1. 文档目标

- 解决“创建了流程模板，但部分流程无法启动”的结构性问题
- 统一单任务与流程任务的执行模型，避免两套编排逻辑长期分叉
- 将“模板”“输入数据”“运行实例”三层彻底拆开
- 让任意流程模板在选择店铺后都能被启动；是否成功执行由预检和任务入参决定，而不是由是否存在 `flow_params` 决定

## 2. 当前问题概述

当前项目中：

- `flows` 表保存流程模板
- `flow_params` 同时承担流程输入数据和流程运行状态
- 单任务模式可直接启动，不依赖 `flow_params`
- 流程模式必须先存在待执行的 `flow_params`

这会带来三个直接问题：

1. 创建了流程模板，不等于这个流程能启动
2. 没有导入流程参数时，流程模式会被静默跳过，表现为“点了启动但没跑”
3. 流程执行编排存在 Worker 侧和主进程侧的双重推进风险

## 3. 改造目标

改造后系统应满足以下行为：

- 创建任意数量的流程模板，都可以启动
- 启动流程时，系统一定先创建“运行实例”
- 有输入数据时，按输入展开执行
- 没有输入数据时，也可按店铺生成空运行项
- 任务是否成功由任务自身的参数要求决定，不由模板是否存在实例记录决定
- Worker 只执行单步，不负责决定下一步
- 单任务与流程最终走同一套执行引擎

## 4. 新架构原则

### 4.1 三层拆分

- 模板层：定义“要跑什么”
- 输入层：定义“带什么数据跑”
- 运行层：定义“这次实际怎么跑”

### 4.2 单任务与流程统一

- 单任务视为只有 1 步的流程
- 对外可以保留单任务入口
- 对内统一转换成运行实例

### 4.3 启动与校验分离

- 启动流程：永远允许创建运行实例
- 执行校验：在预检或步骤执行前判断参数是否完整
- 缺参时给出明确失败原因，而不是不启动

### 4.4 编排职责唯一

- 主进程负责：
  - 创建运行实例
  - 预检
  - barrier / merge / retry / on_fail 编排
  - 状态推进
  - SSE 推送
  - 回调
- Worker 负责：
  - 接收单步任务
  - 执行任务
  - 回传结果

## 5. 数据模型设计

## 5.1 保留表：`flows`

继续保存流程模板，结构基本沿用现有设计。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT PK | 流程 ID |
| `name` | TEXT | 流程名称 |
| `steps` | TEXT(JSON) | 步骤定义 |
| `description` | TEXT | 流程说明 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

`steps` 建议继续使用以下结构：

```json
[
  {
    "task": "登录",
    "on_fail": "continue",
    "barrier": false,
    "merge": false
  }
]
```

## 5.2 新增表：`flow_input_sets`

表示某个流程的一组可复用输入数据。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT PK | 输入集 ID |
| `flow_id` | TEXT | 关联流程 ID |
| `name` | TEXT | 输入集名称 |
| `description` | TEXT | 描述 |
| `source_type` | TEXT | `manual/csv/xlsx/api` |
| `enabled` | INTEGER | 是否启用 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

建议索引：

- `idx_flow_input_sets_flow_id(flow_id)`

## 5.3 新增表：`flow_input_rows`

表示输入集中的具体数据行。只表达输入，不存运行状态。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PK | 行 ID |
| `input_set_id` | TEXT | 关联输入集 |
| `shop_id` | TEXT | 店铺 ID |
| `input_data` | TEXT(JSON) | 输入参数 |
| `enabled` | INTEGER | 是否启用 |
| `sort_order` | INTEGER | 排序 |
| `source_key` | TEXT | 幂等去重键，可选 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

建议索引：

- `idx_flow_input_rows_set_shop(input_set_id, shop_id)`
- `idx_flow_input_rows_source_key(source_key)`

## 5.4 新增表：`execution_runs`

表示一次“启动流程”或“启动单任务”的总运行记录。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT PK | 运行 ID |
| `mode` | TEXT | `flow/task` |
| `flow_id` | TEXT NULL | 流程模式使用 |
| `task_name` | TEXT NULL | 单任务模式使用 |
| `flow_snapshot` | TEXT(JSON) | 启动时冻结的步骤快照 |
| `input_set_id` | TEXT NULL | 使用的输入集 |
| `shop_ids` | TEXT(JSON) | 本次目标店铺 |
| `requested_concurrency` | INTEGER | 请求并发 |
| `actual_concurrency` | INTEGER | 实际调度并发 |
| `callback_url` | TEXT NULL | 回调地址 |
| `trigger_source` | TEXT | `manual/schedule/api/agent` |
| `status` | TEXT | `queued/running/completed/partial_failed/failed/cancelled` |
| `total_items` | INTEGER | 总执行项 |
| `queued_items` | INTEGER | 排队中 |
| `running_items` | INTEGER | 运行中 |
| `success_items` | INTEGER | 成功数 |
| `failed_items` | INTEGER | 失败数 |
| `cancelled_items` | INTEGER | 取消数 |
| `error` | TEXT NULL | 批次级错误 |
| `created_at` | DATETIME | 创建时间 |
| `started_at` | DATETIME NULL | 开始时间 |
| `finished_at` | DATETIME NULL | 结束时间 |
| `updated_at` | DATETIME | 更新时间 |

建议索引：

- `idx_execution_runs_status(status)`
- `idx_execution_runs_flow_id(flow_id)`
- `idx_execution_runs_created_at(created_at DESC)`

## 5.5 新增表：`execution_run_items`

表示一次运行下的实际执行单元。每个店铺、每条输入展开后各生成一条。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT PK | 执行项 ID |
| `run_id` | TEXT | 所属运行 |
| `shop_id` | TEXT | 店铺 ID |
| `input_row_id` | INTEGER NULL | 对应输入行 |
| `context_data` | TEXT(JSON) | 初始上下文 |
| `current_step` | INTEGER | 当前步骤 |
| `total_steps` | INTEGER | 总步骤数 |
| `status` | TEXT | `queued/precheck_failed/waiting_barrier/running/success/failed/skipped/cancelled` |
| `last_task_name` | TEXT NULL | 最近一步任务 |
| `error` | TEXT NULL | 错误信息 |
| `result_data` | TEXT(JSON) | 最终结果摘要 |
| `created_at` | DATETIME | 创建时间 |
| `started_at` | DATETIME NULL | 开始时间 |
| `finished_at` | DATETIME NULL | 结束时间 |
| `updated_at` | DATETIME | 更新时间 |

建议索引：

- `idx_execution_run_items_run_id(run_id)`
- `idx_execution_run_items_shop(run_id, shop_id)`
- `idx_execution_run_items_status(run_id, status)`

## 5.6 新增表：`execution_run_steps`

表示每个执行项的逐步执行记录，用于替代 `flow_params.step_results` 的运行职责。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT PK | 步骤记录 ID |
| `run_item_id` | TEXT | 所属执行项 |
| `step_index` | INTEGER | 第几步 |
| `task_name` | TEXT | 任务名 |
| `on_fail` | TEXT | 失败策略 |
| `barrier` | INTEGER | 是否屏障 |
| `merge` | INTEGER | 是否合并 |
| `status` | TEXT | `queued/waiting_barrier/running/success/failed/skipped/cancelled` |
| `params_snapshot` | TEXT(JSON) | 本步入参快照 |
| `result_data` | TEXT(JSON) | 本步结果 |
| `error` | TEXT NULL | 错误信息 |
| `celery_task_id` | TEXT NULL | Celery 任务 ID |
| `worker_id` | TEXT NULL | 执行 Worker / 机器标识 |
| `created_at` | DATETIME | 创建时间 |
| `started_at` | DATETIME NULL | 开始时间 |
| `finished_at` | DATETIME NULL | 结束时间 |
| `updated_at` | DATETIME | 更新时间 |

建议索引：

- `idx_execution_run_steps_item(run_item_id, step_index)`
- `idx_execution_run_steps_celery(celery_task_id)`

## 5.7 旧表兼容建议

### `flow_params`

不建议继续承担运行状态职责，建议逐步迁移为历史兼容或只读兼容来源：

- `shop_id` -> `flow_input_rows.shop_id`
- `flow_id` -> `flow_input_sets.flow_id`
- `params` -> `flow_input_rows.input_data`
- `enabled` -> `flow_input_rows.enabled`
- `current_step` / `step_results` / `status` / `error` / `batch_id` / `run_count`
  - 迁入 `execution_runs / execution_run_items / execution_run_steps`

### `task_params`

短期可保留，中期建议也统一到“输入 + 运行实例”模型，避免单任务和流程长期双轨。

## 6. 启动规则设计

### 6.1 启动流程

无论是否存在输入集，点击启动都必须先创建一条 `execution_runs` 记录。

### 6.2 运行项展开规则

- 如果指定了 `input_set_id`
  - 按输入行展开 `execution_run_items`
- 如果未指定 `input_set_id`
  - 按店铺展开空 `execution_run_items`

因此只要：

- 流程模板存在
- 选择了店铺

就一定能启动。

### 6.3 预检规则

启动成功不等于业务必定成功。

建议任务注册表提供任务元数据：

- `requires_input`
- `required_fields`
- `supports_empty_context`

例如：

- `登录`
  - `requires_input = false`
- `发布相似商品`
  - `requires_input = true`
  - `required_fields = ["parent_product_id"]`

如果缺少必要字段：

- 允许创建运行实例
- 在预检阶段或步骤开始前标记为 `precheck_failed`
- 错误信息明确写入运行项和步骤记录

## 7. 接口设计草案

所有接口继续沿用项目统一响应：

```json
{
  "code": 0,
  "data": {},
  "msg": "ok"
}
```

## 7.1 流程模板接口

### `GET /api/flows`

获取流程模板列表。

### `POST /api/flows`

创建流程模板。

### `PUT /api/flows/{flow_id}`

更新流程模板。

### `DELETE /api/flows/{flow_id}`

删除流程模板。

### `POST /api/flows/{flow_id}/precheck`

启动前预检。

请求示例：

```json
{
  "shop_ids": ["shop-1"],
  "input_set_id": "input-set-1"
}
```

响应示例：

```json
{
  "code": 0,
  "data": {
    "ok": false,
    "summary": {
      "total_items": 1,
      "pass_items": 0,
      "failed_items": 1
    },
    "items": [
      {
        "shop_id": "shop-1",
        "step_index": 1,
        "task_name": "发布相似商品",
        "ok": false,
        "error": "缺少 parent_product_id"
      }
    ]
  },
  "msg": "预检完成"
}
```

## 7.2 输入集接口

### `GET /api/flows/{flow_id}/input-sets`

获取某个流程的输入集列表。

### `POST /api/flows/{flow_id}/input-sets`

创建输入集。

### `PUT /api/flows/{flow_id}/input-sets/{input_set_id}`

更新输入集。

### `DELETE /api/flows/{flow_id}/input-sets/{input_set_id}`

删除输入集。

### `GET /api/input-sets/{input_set_id}/rows`

获取输入行列表。

### `POST /api/input-sets/{input_set_id}/rows/import`

导入 CSV / XLSX 输入行。

### `POST /api/input-sets/{input_set_id}/rows`

创建单条输入行。

### `PUT /api/input-sets/{input_set_id}/rows/{row_id}`

更新输入行。

### `DELETE /api/input-sets/{input_set_id}/rows/{row_id}`

删除输入行。

## 7.3 流程启动接口

### `POST /api/flows/{flow_id}/runs`

创建一次流程运行实例。

请求示例：

```json
{
  "shop_ids": ["shop-1", "shop-2"],
  "input_set_id": "input-set-1",
  "requested_concurrency": 2,
  "callback_url": "http://127.0.0.1:8001/api/batch-callback",
  "empty_run_policy": "allow_empty"
}
```

响应示例：

```json
{
  "code": 0,
  "data": {
    "run_id": "run-20260325-001",
    "status": "queued",
    "total_items": 6
  },
  "msg": "流程已启动"
}
```

### 空运行策略建议

- `allow_empty`
  - 没有输入行时按店铺生成空运行项
- `require_input`
  - 没有输入行时返回明确错误

建议默认使用 `allow_empty`，以满足“所有流程都能启动”的目标。

## 7.4 运行查询接口

### `GET /api/runs`

分页获取运行列表。

### `GET /api/runs/{run_id}`

获取运行详情。

### `GET /api/runs/{run_id}/items`

获取运行项列表。

### `GET /api/runs/{run_id}/steps`

获取步骤明细。

### `GET /api/runs/{run_id}/events`

SSE 推送运行状态。

### `POST /api/runs/{run_id}/cancel`

取消运行。

### `POST /api/runs/{run_id}/retry-failed`

重试失败项。

## 7.5 单任务统一入口

### `POST /api/tasks/{task_name}/runs`

对外保留单任务入口，但内部直接转换为单步流程运行实例。

请求示例：

```json
{
  "shop_ids": ["shop-1"],
  "requested_concurrency": 1
}
```

内部处理建议：

- 自动生成一个只包含 1 个步骤的 `flow_snapshot`
- 写入 `execution_runs`
- 之后与流程模式共用同一套调度逻辑

## 7.6 旧接口兼容建议

可在过渡期保留以下接口，但内部统一转发到新运行模型：

- `POST /api/execute/batch`
- `POST /api/tasks/execute`

兼容策略：

- 单任务入口 -> 转发到 `POST /api/tasks/{task_name}/runs`
- 流程入口 -> 转发到 `POST /api/flows/{flow_id}/runs`

## 8. 调度与状态推进设计

## 8.1 主进程职责

- 创建运行记录
- 生成运行项和步骤记录
- 执行预检
- 维护 barrier / merge / retry / on_fail 状态机
- 处理批次取消
- 推送 SSE
- 回调外部系统

## 8.2 Worker 职责

- 接收单步执行请求
- 执行业务任务
- 回传结果

Worker 不应决定：

- 下一步是否投递
- 哪些运行项进入 barrier
- merge 的主记录是谁

## 8.3 Worker 入参建议

建议将 Worker 入参最小化为：

```json
{
  "run_step_id": "step-001"
}
```

这样运行编排只能由主进程完成，避免出现双重推进。

## 9. 并发设计建议

当前 `concurrency` 不应只作为记录用途，而应转为真实调度能力。

建议规则：

- 所有运行项先进入 `queued`
- 调度器按以下维度拉起：
  - 全局并发上限
  - 店铺级串行锁
  - 机器级资源限制
  - Worker 可用性

这样系统可以做到：

- 创建很多流程都能启动
- 不是所有流程都必须立刻执行
- 允许排队、暂停、取消和重试

## 10. 迁移策略建议

建议分阶段迁移：

### Phase 1：新增运行实例层

- 新增 `execution_runs`
- 新增 `execution_run_items`
- 新增 `execution_run_steps`
- 不删旧表

### Phase 2：流程启动切到新模型

- `POST /api/execute/batch` 内部切到 `runs`
- 流程模式不再依赖 `flow_params` 是否存在

### Phase 3：输入层拆分

- 新增 `flow_input_sets`
- 新增 `flow_input_rows`
- 将 `flow_params` 的输入职责迁出

### Phase 4：单任务统一

- `task_params` 也逐步接入 `runs`
- 单任务与流程共用同一执行引擎

### Phase 5：下线旧运行状态依赖

- `flow_params.step_results` 停止作为真实运行状态来源
- 旧接口转为兼容适配层

## 11. 对前端的影响

前端页面建议拆成三块：

- 流程模板管理
- 输入集管理
- 运行中心

运行中心只关心：

- 选流程
- 选店铺
- 选输入集
- 点击启动
- 查看 run / run_item / step 状态

这样用户不会再把“模板”和“实例”混淆。

## 12. 预期收益

完成改造后：

- 任意流程模板都可以被启动
- “没有输入所以流程没跑”会变成明确可见的预检失败或空运行策略
- 单任务与流程只保留一套执行引擎
- 调度、重试、取消、回调、SSE 的行为更加一致
- Worker 与主进程的职责边界清晰，不再双重推进

## 13. 一句话结论

当前项目缺少的是“运行实例层”。

只要把：

- 模板：`flows`
- 输入：`flow_input_sets / flow_input_rows`
- 运行：`execution_runs / execution_run_items / execution_run_steps`

这三层拆开，流程数量就不再是问题，系统也能真正做到“创建多少流程，都可以启动”。
