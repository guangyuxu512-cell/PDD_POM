# 测试策略

## 1. 测试框架

- `pytest`
- `pytest-asyncio`

## 2. 配置文件

- 测试配置位于 `tests/pytest.ini`
- 当前关键配置：
  - `asyncio_mode = auto`
  - `python_files = test_*.py 测试_*.py`
  - `python_classes = Test* 测试_*`

## 3. 当前测试目录

```text
tests/
├─ 单元测试/
├─ conftest.py
└─ pytest.ini
```

- 当前仓库主要测试集中在 `tests/单元测试/`
- `tests/集成测试/`：当前项目暂无此内容

- `tests/单元测试/测试_基础页.py`
- `tests/单元测试/测试_登录任务.py`
- `tests/单元测试/测试_验证码识别.py`

## 5. 覆盖范围

- 页面基类安全操作
- 登录任务执行
- 验证码识别与反检测
- Celery 桥接与 Worker 初始化
- FastAPI 启动入口
- 系统接口
- 浏览器服务
- 心跳服务
- 任务服务统一入口

## 6. 测试命名与风格

- 测试文件命名遵循：
  - `test_*.py`
  - `测试_*.py`
- 当前仓库主要采用中文测试文件名。
- 测试应优先覆盖：
  - 浏览器流程
  - 任务执行
  - 接口服务
  - Worker 桥接
- 新增功能或修复缺陷时，优先补针对性回归测试。

## 7. 当前测试策略总结

- 当前项目以单元测试和回归测试为主。
- 大量测试围绕服务层、启动入口、桥接逻辑和页面基类展开。
- 现阶段未看到独立的端到端集成测试目录或自动化部署测试流程。

## 8. 常用命令

- 运行全部测试：

```bash
python -m pytest -c tests/pytest.ini -q
```

- 当前仓库最近一轮测试记录为：

```text
29 passed
```

## 9. 与规则文件的关系

- `.claude/validator.md` 强调测试与审查时重点检查：
  - 是否直接调用 `page.click()` 而非安全方法
  - `POM` 是否混入业务逻辑
  - `Task` 是否缺少 `@自动回调`
  - API 是否使用统一响应封装
  - 后端是否误用英文命名
  - 前端是否误用中文命名

- 独立集成测试目录：当前项目暂无此内容
